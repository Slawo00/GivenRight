## 2️⃣ Replit – Fetch Pattern (Screen 4)

```ts
// lib/loadScreen4Options.ts
import { supabase } from './supabase';

export async function loadScreen4Options() {
  const [giftType, time] = await Promise.all([
    supabase
      .from('q_gift_type_preferences')
      .select('code,label')
      .order('sort_order'),

    supabase
      .from('q_time_constraints')
      .select('code,label')
      .order('urgency_level')
  ]);

  if (giftType.error || time.error) {
    throw new Error('Failed to load Screen 4 options');
  }

  return {
    giftTypes: giftType.data,
    timeConstraints: time.data
  };
}
```

## 5️⃣ Persistenz (empfohlenes Muster)

👉 **Sauberste Lösung:** eigene Tabelle
(Trennung von Beziehung, Person & Constraints)

### Tabelle: `decision_parameters`

```sql
CREATE TABLE decision_parameters (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  relationship_id UUID REFERENCES relationships(relationship_id) ON DELETE CASCADE,

  gift_type TEXT NOT NULL,
  time_constraint TEXT NOT NULL,

  created_at TIMESTAMPTZ DEFAULT now()
);
```

### Update/Insert aus Replit

```ts
// services/saveDecisionParameters.ts
import { supabase } from './supabase';

export async function saveDecisionParameters(
  relationshipId: string,
  data: { gift_type: string; time_constraint: string }
) {
  const { error } = await supabase
    .from('decision_parameters')
    .upsert({
      relationship_id: relationshipId,
      gift_type: data.gift_type,
      time_constraint: data.time_constraint
    });

  if (error) throw error;
}
```

> Falls ihr **bewusst** alles in `recipients` halten wollt: technisch möglich,
> aber langfristig weniger sauber. Meine Empfehlung bleibt: **separate Tabelle**.
